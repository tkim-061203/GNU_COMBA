            COMBA-PROMPT LangGraph Core — Framework Overview

  Project:  GNU_COMBA / langgraph_core

  Purpose:  LLM-driven automated Verilog generation, verification, and
            iterative debugging using LangGraph state machines.

1. WHAT IS THIS?
================================================================================

  langgraph_core is the central pipeline engine of the GNU_COMBA thesis project.
  It takes a natural-language (NL) description of a digital hardware module,
  converts it to structured XML, generates synthesizable Verilog code using an
  LLM, and then iteratively compiles, simulates, and debugs the output until the
  design passes all tests — or a configurable trial limit is reached.

  The pipeline is implemented as a LangGraph state machine with conditional
  routing, enabling multi-loop correction cycles with escalating prompt
  strategies, do-no-harm rollback guards, and hierarchical best-of-N sampling.

3. PIPELINE ARCHITECTURE
================================================================================

  3.1 High-Level Flow
  ────────────────────

    NL Text ──► Converter ──► COMBA XML ──► Generator ──► Raw LLM Output
                                                              │
                                                              ▼
                                                         Sanitizer
                                                              │
                                                              ▼
                                              ┌──────── Syntax Check (SC)
                                              │               │
                                         [FAIL]          [PASS]
                                              │               │
                                              ▼               ▼
                                          Guard SC       TB Simulation (TS)
                                              │               │
                                    [ROLLBACK] [OK]     [FAIL]    [PASS]
                                        │       │          │         │
                                        │       ▼          ▼         ▼
                                        │   TED_SC     Guard TS    END ✅
                                        │       │          │
                                        │       ▼     [ROLLBACK] [OK]
                                        │   Debugger       │      │
                                        │       │          │      ▼
                                        │       ▼          │   TED_TB
                                        │   Sanitizer      │      │
                                        │       │          │      ▼
                                        └───────┴──────────┘   Debugger
                                                                  │
                                                                  ▼
                                                              Sanitizer
                                                                  │
                                                              (loop back
                                                               to SC)

  3.2 Node Descriptions (v4 — 11 nodes)
  ──────────────────────────────────────

    1. node_converter      NL → COMBA XML. Validates schema with pydantic-xml.
                           Retries up to xml_retry_limit (default 2).

    2. node_generator      XML + Original Spec → raw Verilog. Uses base model.
                           Initializes MultiAttemptManager.

    3. node_sanitizer      Cleans raw LLM output: strips fences, XML tags,
                           prose lines; extracts module block; auto-promotes
                           output ports to reg when needed.

    4. node_syntax_check   Runs iverilog -tnull (lint-only) on sanitized code.
                           Counts genuine error lines. Captures baseline SC
                           count for guard comparison.

    5. node_guard_sc       Do-no-harm guard (SC phase). Compares post-debugger
                           candidate against pre-debugger snapshot. Rolls back
                           GVD if candidate has more errors. Tracks bad_streak.

    6. node_ted_syntax     Extracts topmost compiler error from SC log.
                           Updates EDTM counter. Prepares EDP (Exception
                           Debugging Prompt) with optional escalation warning.

    7. node_debugger       Invokes LLM (LoRA model) to fix the code. Snapshots
                           current state before mutation. Delegates prompt
                           building to MultiAttemptManager for L0-L4 escalation.

    8. node_tb_sim         If SC passes: compiles design + testbench with
                           iverilog and runs simulation. Checks for assertion
                           failures and trace mismatches.

    9. node_guard_ts       Do-no-harm guard (TS phase). Same rollback logic as
                           guard_sc but for testbench results.

   10. node_ted_tb         Extracts topmost TB failure. Formulates TDP
                           (Testbench Debugging Prompt).

   11. node_classify_tb    (v5) Classifies TB failure type: fsm_state_error,
       + node_vcd_analyzer timing_error, timeout, output_logic, etc.
                           Parses VCD waveforms for state trace hints.


  3.3 State Schema (COMBAState)
  ─────────────────────────────

    The pipeline state is a TypedDict with ~45 fields organized into groups:

      Input/Output:       nl_input, xml_description, xml_valid, module_name,
                          benchmark_id, xml_retry_count/limit.

      Generated Verilog:  gvd (current code), sgvd (snapshot), _raw_llm_output.

      Syntax Check:       sc_log, sc_exception, sc_exception_count,
                          sc_prev_exception_count.

      Testbench:          tb_log, tb_failure.

      Debugging Prompts:  edp (syntax), tdp (functional).

      Control:            edtm (error tracking dict), phase (sc/ts),
                          sc_trial, ts_trial, total_iter, rollback_triggered.

      Guard:              guard_prev_gvd, guard_prev_sc_count,
                          guard_baseline_gvd, guard_bad_streak,
                          guard_total_rollbacks, guard_total_commits.

      MultiAttempt:       multi_attempt_mgr, escalation_level, _last_llm_source.

      Result:             final_status, error, dataset_dir, work_dir,
                          expected_header.


  3.4 Conditional Routing
  ───────────────────────

    7 conditional edges govern the flow:

      route_after_converter   → generator | converter (retry) | END
      route_after_sanitizer   → syntax_check | generator (re-query)
      route_after_guard_sc    → tb_sim (pass) | ted_syntax (fail) | END (limit)
      route_after_ted_sc      → debugger | END (EDTM exhausted)
      route_after_guard_ts    → END (pass) | classify_tb (fail) | END (limit)
      route_after_classify_tb → vcd_analyzer | ted_tb (direct)
      route_after_ted_tb      → debugger | END (EDTM exhausted)


4. KEY MECHANISMS
================================================================================

  4.1 EDTM (Exception-Debugging Trial Management)
  ─────────────────────────────────────────────────

    Tracks the exact signature of each compiler/testbench error across loop
    iterations. Error keys are normalized (line numbers stripped, whitespace
    collapsed, truncated to 100 chars). If the same error is seen more than
    EDTM_MAX_RETRIES (default 3) times, the prompt forcefully flags it as
    unresolvable and the escalation level increases.

  4.2 Do-No-Harm Guards
  ─────────────────────

    Before the debugger node runs, the current state is snapshotted
    (guard_prev_gvd, guard_prev_sc_count). After the debugger produces a
    candidate fix:

      - If the candidate has MORE errors than the snapshot → ROLLBACK
      - If the candidate is equal or better → COMMIT

    A bad_streak counter increments on consecutive rollbacks. If
    bad_streak >= GUARD_MAX_BAD_STREAK (default 2), the pipeline terminates
    and restores the generator's baseline GVD if it scores better.

  4.3 Multi-Attempt Escalation (L0 → L4)
  ───────────────────────────────────────

    The MultiAttemptManager tracks attempts per error_key and progressively
    increases prompt strictness:

      L0 (Standard):      Normal EDP/TDP prompt.
      L1 (With History):  Appends "your previous fix also failed" context.
      L2 (Hint):          Injects category-specific design hints (40+ modules)
                          plus error-pattern-specific guidance.
      L3 (Rethink):       Forces complete architectural rethink. Extracts
                          the fixed header, asks LLM to redesign from scratch.
      L4 (Skeleton):      Provides bare module skeleton (ports + declarations),
                          asks LLM to fill only the behavioral logic.

    After 5 attempts on the same error, should_give_up() returns True.

  4.4 Hierarchical Self-Consistency (Best-of-N)
  ─────────────────────────────────────────────

    Enabled via COMBA_SELF_CONSISTENCY=1 environment variable.

      Tier 1 (Deterministic):  Sample 0 at T=0.0. If it passes, return
                               immediately (1× cost, ~60% of cases).

      Tier 2 (Diversity):      Samples 1..N-1 at increasing temperatures
                               (0.5, 0.5, 0.8, 0.8, 1.0, ...).
                               Each sample gets a unique "Diversity Hint"
                               injected into the generator system prompt.

      Scoring:  (status_rank, -tb_errors, -sc_errors, -code_size, -sample_idx)
                Higher = better. Tie-break favors earlier sample (determinism).

      Early Exit:  Pipeline stops on first PASS sample.
      Wall Budget: Optional time limit (COMBA_WALL_BUDGET env var).

  4.5 Verilog Sanitizer (8-stage pipeline)
  ────────────────────────────────────────

    S1: Strip markdown fences (```verilog, ```, etc.)
    S2: Normalize LLM hallucinations (e.g., "dmodule" → "endmodule")
        Strip known COMBA XML tags and HTML comments.
    S3: Strip Verilog comments (line // and block /* */)
    S4: Extract module..endmodule block (robust multi-module handling)
    S5: Strip prose lines (lines without Verilog syntax)
    S6: Normalize whitespace (collapse triple+ blank lines)
    S7: Structural checks: empty logic, placeholders, reg promotion,
        missing default in case, duplicate modules, high-impedance,
        FSM lag, mixed clock edges, port count mismatch.
    S8: Auto-repair: promote output ports to 'reg' when assigned in
        always blocks.

  4.6 FSM-Aware TB Debugging (v5 Patch)
  ──────────────────────────────────────

    The v5 patch adds intelligent failure classification:

      Classifier:   Categorizes TB failures into: timeout, fsm_state_error,
                    timing_error, output_logic, combinational_mismatch.
                    Uses both error text heuristics and module name hints.

      VCD Analyzer: For FSM/timing failures, parses the dump.vcd waveform
                    file to extract state register transitions. Produces a
                    compact hint like:
                      [VCD STATE TRACE]
                        tb.dut.state: t=0:00, t=10:01, t=20:10, t=30:11
                      [FAILURE_CYCLE] simtime=35

      Structured TDP:  Combines failure type, failure content, trace,
                       VCD hints, and root-cause suggestions into a
                       machine-readable format for the debugger LLM.


5. LLM INTERFACE & ROUTING
================================================================================

  COMBALlm (llm_interface.py) wraps dual-GPU vLLM endpoints as a LangChain
  BaseChatModel:

    Server Modes:
      - "dual"     :  2 vLLM servers (GPU0 for base, GPU1 for LoRA debugger)
      - "single"   :  1 vLLM server with LoRA, same client for both
      - "langchain" : LangChain ChatOpenAI wrapper (dev/laptop mode)

    Routing:
      switch_to_base()  → Converter + Generator  → base_url (:8000)
      switch_to_lora()  → Debugger               → debugger_url (:8001)

    Direct API:
      call_llm_base(messages)  → text response from base model
      call_llm_lora(messages)  → text response from LoRA model

    Factory:  COMBALlm.from_env() reads .env for URLs, API keys, model names.
    Fallback: COMBALlm.from_langchain(llm) wraps any LangChain model.

    Retry:    Exponential backoff (max 3 retries) on transient errors.
    Health:   health_check() queries both servers for available models.


6. PROMPT ENGINEERING
================================================================================

  All prompts are centralized in prompts.py.

  6.1 Converter Prompt (CONVERTER_SYSTEM_PROMPT)
  ──────────────────────────────────────────────

    Rules:
      - Module name MUST exactly match the spec (no rename/abbreviation).
      - Self-contained: "instantiate X" → describe inline behavior.
      - parameter_description: ONLY FSM states and named constants.
      - Do NOT hallucinate internal registers/wires in XML.
    Includes 2 few-shot examples (combinational adder + sequential counter).

  6.2 Generator Prompt (GENERATOR_SYSTEM_PROMPT)
  ──────────────────────────────────────────────

    9 critical rules (R1-R9):
      R1: Self-contained — no external sub-modules.
      R2: Assignment discipline (= for comb, <= for seq).
      R3: Indexed part-select (signal[N*i +: W]) instead of variable range.
      R4: MIPS shift/ALU conventions.
      R5: Combinational multi-step algorithms (for-loop with blocking =).
      R6: FSM sequence detection (overlapping, prefix matching).
      R7: Timer/counter FSM patterns.
      R8: Honor exact enum/parameter values from XML.
      R9: Port name fidelity — never add/rename/drop.

  6.3 EDP Prompt (Syntax Debugging)
  ─────────────────────────────────

    Error → Fix mapping for 12 common iverilog/Verilator errors.
    Pattern-based constraint injection: detects error categories in logs
    and appends specific fix instructions.

  6.4 TDP Prompt (Functional Debugging)
  ─────────────────────────────────────

    Debugging methodology: compare OUTPUT vs REFERENCE traces, trace backward,
    check 12 common root causes (operand swap, off-by-one, reset polarity,
    accumulator timing, unused inputs, FSM transitions, wrong operator, width
    truncation).

    Pattern-specific rules: P1 (unused port), P2 (shift via concatenation),
    P3 (counter/handshake logic).

    Hint injection: automatically detects patterns in SC log and code,
    appends targeted fix hints.


7. EXECUTION & SERVING
================================================================================

  7.1 Pipeline Runner (pipeline_runner.py)
  ────────────────────────────────────────

    LLM Factory Chain:
      1. COMBA_USE_STUB=1 → StubLLM (testing)
      2. COMBALlm.from_env() (dual-GPU vLLM)
      3. ChatOpenAI (Ollama / single vLLM fallback)

    Entry Points:
      run_pipeline_sync(nl_input, ...)
        → Single module run. Auto-dispatches to multi_sample if SC=1.

      run_pipeline_streaming(nl_input, ...)
        → Yields (node_name, state_update) per step. For UI/API.

      run_pipeline_batch(module_paths, ...)
        → Batch evaluation over directories. Generates:
            - Per-module JSON reports (reports/report_langgraph.{type}.json)
            - Global JSON summary (reports/summary_langgraph.{type}.json)
            - Markdown summary table (reports/summary_langgraph.{type}.md)

  7.2 API Server (api_server.py)
  ──────────────────────────────

    FastAPI app exposing:
      GET  /v1/models                → list available models
      POST /v1/chat/completions      → run pipeline (streaming or non-streaming)
      GET  /health                   → basic health check
      GET  /health/llm               → detailed LLM server health

    OpenAI-compatible: works with Open WebUI, any OpenAI SDK client.
    SSE streaming: each pipeline node emits a formatted event with emoji
    status indicators.

  7.3 CLI App (langgraph_app.py)
  ─────────────────────────────

    Simplified 3-node graph: Converter → Human Review → Generator.
    Usage:
      python langgraph_app.py "Design an 8-bit adder"
      python langgraph_app.py --xml modules/adder/desc.xml
      python langgraph_app.py --no-review "Design a 4-bit counter"
      python langgraph_app.py -o output.v "Design a frequency divider"

9. TESTING
================================================================================

  Three test suites (all pytest-based):

    test_pipeline.py       Full pipeline integration tests:
                           - Happy path (stub returns correct code)
                           - Buggy path (stub returns buggy, then fixed)
                           - Always-buggy (iteration limit reached)
                           - Rollback test (fix makes things worse)
                           - Streaming mode test
                           - Batch mode test

    test_guard.py          Do-no-harm guard unit tests:
                           - SC rollback on regression
                           - SC commit on improvement
                           - Bad streak termination
                           - Baseline restoration at terminal
                           - TS guard behavior

    test_debugger_model.py Debugger and multi-attempt tests:
                           - Escalation level progression (L0 → L4)
                           - Category hint injection
                           - Skeleton prompt generation
                           - Error-specific constraint injection
                           - should_give_up() behavior

  Run:
    cd src/langgraph_core
    COMBA_USE_STUB=1 COMBA_QUIET=1 python -m pytest -v
